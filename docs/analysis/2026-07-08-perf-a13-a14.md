# a13 → a14 performance: honest before/after across the KVM fleet

**Date:** 2026-07-08 · **Versions:** kvm-pilot `0.1.0a13` → `0.1.0a14` · **Fleet:** GL-RM1PE `.11` / `.20` / `.39`

The first performance baseline. a14 shipped the interface router (#181), persistent
SSH (#182), and dep-free SSH password auth (#183). This records what actually got
faster, in real terms, measured a13-vs-a14 on real hardware — reported honestly so
the next round of perf work starts from a truthful line, not a marketing one.

## Method
- **Library-level warm p50** — the latency an agent / the MCP server actually pays
  (in-process driver calls, cold sample dropped). Measured by a version-agnostic
  harness run under an **isolated a13 venv** (PyPI `0.1.0a13`, Python 3.12) and the
  **a14** build — same harness, same devices, same profiles.
- **Fleet-parallel, equipment-isolated** — one performance-testing agent per KVM ran
  a13 *then* a14 sequentially against **its own** device; no two agents touched the
  same box at once (background workflow `ab-perf-a13-a14`, 3 measure agents + 1
  verifier).
- **Adversarially verified** — a reviewer agent checked every delta against the raw
  numbers with a noise floor (a move under **~15% OR ~20 ms is "flat"**), and was
  told to flag overclaims. It credited only the SSH win, marked two `.20` blips as
  non-creditable variance, and returned **no overclaims** in the final report.
- **Use cases:** KVM-API reads (`info`/`logs`/`snapshot`/`power_state`) + HID
  (`mouse_move`) + an in-band **"run 5 commands over SSH"** task.

## a13 → a14 latency: honest before/after (library-level warm p50, ms)

**Bottom line:** a14's one real, measurable win is **persistent SSH (ControlMaster)**
on the two SSH-capable devices — a **~2.5× speedup** on the "run 5 commands" task.
The **KVM-API ops are flat**, exactly as expected: a14 did not change the HTTP
transport, so `info`/`logs`/`snapshot`/`power_state`/`hid_mouse` land in the same
range on both versions. This is the **first** pass of ongoing perf work; the next
lever is HTTP keep-alive (**#185**), which is what would actually move the KVM-API
numbers.

Discipline: a per-op move is **noise / "flat"** if under ~15% **or** under ~20 ms.
Only SSH runs with `fresh_ok == 5 && persist_ok == 5` count. `.39` has no SSH.

### homelab — 10.0.1.11
| op | a13 ms | a14 ms | delta | verdict |
|---|---|---|---|---|
| info | 202.2 | 175.1 | −27.1 (−13.4%) | flat (<15%) |
| logs | 57.2 | 65.7 | +8.5 (+14.9%) | flat (<20ms) |
| power_state | 56.4 | 58.0 | +1.6 (+2.8%) | flat |
| snapshot | 90.2 | 82.0 | −8.2 (−9.1%) | flat |
| hid_mouse | 48.8 | 68.1 | +19.3 (+39.5%) | flat (<20ms, at the noise ceiling — watch) |

**SSH 5-command task:** a13 persist **1137.1** (≈ its fresh 1084.1 — a13 ignores the
persist flag) → a14 persist **447.2**. **2.54× faster, saves ~690 ms.** (The fresh
5-connection cost is comparable across versions — 1084 vs 1196 — that path is
unchanged.)

### homelab2 — 10.0.1.20
| op | a13 ms | a14 ms | delta | verdict |
|---|---|---|---|---|
| info | 214.3 | 174.4 | −39.9 (−18.6%) | **moved but NOT credited** — transport unchanged; single-sample variance |
| logs | 477.3 | 510.9 | +33.6 (+7.0%) | flat (<15%) |
| power_state | 44.2 | 72.3 | +28.1 (+63.6%) | **moved (worse), NOT blamed on a14** — variance (flat on the other two devices) |
| snapshot | 75.1 | 83.5 | +8.4 (+11.2%) | flat |
| hid_mouse | 67.1 | 60.1 | −7.0 (−10.4%) | flat |

**SSH 5-command task:** a13 persist **1018.0** (≈ its fresh 1191.3) → a14 persist
**405.2**. **2.51× faster, saves ~613 ms.**

|---|---|---|---|---|
| info | 172.3 | 156.3 | −16.0 (−9.3%) | flat |
| logs | 423.4 | 370.5 | −52.9 (−12.5%) | flat (<15%) |

### The SSH win, stated honestly
a13 opens a fresh connection per command (it ignores the persist flag, so its
`persist` number ≈ its `fresh` number — both are full-setup cost). a14 honors
`ControlMaster`, so the 5 commands share one connection.

| device | a13 (fresh-cost) | a14 (persistent) | 5-cmd speedup | saved |
|---|---|---|---|---|
| homelab (.11) | 1137.1 ms | 447.2 ms | **2.54×** | ~690 ms |
| homelab2 (.20) | 1018.0 ms | 405.2 ms | **2.51×** | ~613 ms |

Deliberate caveat: command #1 still pays connection setup even in a14, so the
**per-additional-command** win is larger than the whole-task win. Marginal math —
a13 pays ~200–230 ms *every* command (mostly setup); a14 pays setup once (~200 ms)
then ~50 ms per remaining command → roughly **~4× per extra command**. The 5-command
task, dominated by that unavoidable first setup, therefore shows ~2.5×. Any claim
above ~2.5× *for the task* would be an overclaim.

### What is FLAT, and why
The five KVM-API ops go over the **same HTTP transport** in a13 and a14 — a14 didn't
touch it. 13 of 15 op-deltas across the three devices land inside the noise floor.
The honest statement: **the KVM-API is flat a13→a14.** No KVM-API op got faster
because of a14, and none is advertised as such. The two `.20` deltas that cleared
the thresholds (`info` −18.6%, `power_state` +63.6%) point in opposite directions on
one warm-p50 sample over an unchanged transport — measurement variance, not signal;
an argument for more samples, not a perf claim.

### Framing
The deliverable is narrow and real: **persistent SSH cuts multi-command SSH tasks
~2.5×.** The KVM-API is untouched and honestly flat. The **next lever is HTTP
keep-alive (#185)** — connection reuse on the KVM-API side is the analog of what
ControlMaster just did for SSH, and it's what would move the KVM-API off "flat."

## Reproduce
- Harness: `abharness.py` — `<a13py|a14py> abharness.py <label> <profile> [ssh_host ssh_user ssh_key]`
  (measures the ops above; SSH block runs the 5-command task fresh vs persist —
  a13 silently ignores `persist`, a14 honors it).
- Fan-out: workflow `ab-perf-a13-a14` (one agent per device, adversarial verify).
